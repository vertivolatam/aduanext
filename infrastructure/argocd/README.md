# ArgoCD — GitOps for AduaNext

ArgoCD watches `infrastructure/helm-charts/aduanext/` on `main` and keeps the `aduanext` namespace in sync. Any push to `main` that changes a Helm value or template is auto-applied within ~3 minutes.

## Why GitOps

- **Single source of truth**: the cluster state is whatever `main` says it is. No manual `helm upgrade` drift.
- **Auditability**: every production change is a merged PR with code review, CI checks, and a signed commit.
- **Reproducibility**: a fresh Minikube cluster can be brought to production parity with `make argocd-install && make argocd-app-create`.

## Components

| File | Purpose |
|------|---------|
| `install.yaml` | ArgoCD Namespace — applied before the kustomize bundle |
| `kustomization.yaml` | Pins ArgoCD v2.13.3 upstream install manifest |
| `app-project.yaml` | `AppProject` restricting sources + destinations |
| `application.yaml` | `Application` pointing at `infrastructure/helm-charts/aduanext` |

## Setup (first time)

Requires a running Minikube cluster (`make minikube-up` from VRTV-56).

```bash
# 1. Install ArgoCD v2.13.3
make argocd-install

# 2. Wait for argocd-server to become Ready (happens inside the target)
#    — or check manually:
kubectl get pods -n argocd

# 3. Get initial admin password (autogenerated by ArgoCD)
make argocd-admin-password
# -> prints a bcrypt-ed password; username is `admin`

# 4. Port-forward the UI
make argocd-port-forward
# -> ArgoCD UI at https://localhost:8081 (self-signed cert)

# 5. Create the AppProject + Application (bootstraps GitOps)
make argocd-app-create

# 6. Watch the initial sync
kubectl get application -n argocd -w
```

## Migrating from manual `make helm-install` to GitOps

Before this issue, developers ran `make helm-install` by hand (VRTV-56). After ArgoCD is installed, **do not** run `helm install` against the `aduanext` release anymore — ArgoCD owns it.

To switch:

```bash
# If you ran make helm-install earlier, uninstall the manual release first:
helm uninstall aduanext -n aduanext

# Then hand the namespace over to ArgoCD:
make argocd-app-create
```

## Forcing a sync

ArgoCD auto-syncs on poll (~3m). To force an immediate sync (e.g. after a hotfix):

```bash
# Via the argocd CLI (install with `brew install argocd` or similar):
make argocd-sync

# Or via the UI:
# -> open ArgoCD UI -> aduanext app -> SYNC button
```

## Verifying GitOps end-to-end

1. Edit `infrastructure/helm-charts/aduanext/values.yaml` (e.g. bump `postgresql.primary.resources.requests.cpu` from `100m` to `150m`)
2. Commit + push to `main`
3. Within 3 minutes, `kubectl describe pod -n aduanext aduanext-postgresql-0` shows the new resource request
4. ArgoCD UI shows the Application as `Synced` with the new commit SHA

## Upgrading ArgoCD

1. Bump the version in `kustomization.yaml`:
   ```yaml
   resources:
     - https://raw.githubusercontent.com/argoproj/argo-cd/vX.Y.Z/manifests/install.yaml
   ```
2. `kubectl apply -k infrastructure/argocd/`
3. Watch the rolling upgrade: `kubectl rollout status -n argocd deploy/argocd-server`
4. Smoke test: UI still loads, the `aduanext` app is still `Synced`.

## Troubleshooting

- **Sync stuck on "Progressing"**: `kubectl describe application aduanext -n argocd` shows the specific child resource that is failing.
- **Authentication fails**: retrieve the current admin password — it rotates when the `argocd-initial-admin-secret` is recreated.
- **Drift after a `kubectl edit`**: self-heal reverts the edit within ~3 min. Make the change in git instead.
